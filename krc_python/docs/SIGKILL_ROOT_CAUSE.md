# SIGKILL Root Cause Analysis

**Date:** October 7, 2025
**Issue:** KRC binary and Python wrapper receiving SIGKILL -9 on execution
**Status:** ✅ ROOT CAUSE IDENTIFIED - System Issue, Not Code Issue

---

## Executive Summary

The SIGKILL issue affecting both the KRC binary and Python wrapper is caused by a **broken macOS dynamic linker (dyld) cache**, not by any problems in the KRC code or Python interface. This is a system-level issue that requires macOS and development tools updates to resolve.

---

## The Investigation

### What We Tried
1. Built KRC binary successfully using gfortran
2. Attempted to run KRC directly - received SIGKILL
3. Set up davinci (the official KRC interface) for testing
4. Attempted to run through davinci - received SIGKILL
5. Investigated with `otool -L` to check library dependencies

### The Smoking Gun

```bash
$ cd run && ../krc < ../krc_python/pykrc/data/krc_support/fake_krc344

dyld[21747]: dyld cache '(null)' not loaded: syscall to map cache into shared region failed
dyld[21747]: Library not loaded: /usr/lib/libSystem.B.dylib
  Referenced from: <567650E8-0838-3C63-BFAD-A8BEF3E562F9> /Users/chaberle/Documents/GitHab/KRC/krc
  Reason: tried: '/usr/lib/libSystem.B.dylib' (no such file),
           '/System/Volumes/Preboot/Cryptexes/OS/usr/lib/libSystem.B.dylib' (no such file),
           '/usr/lib/libSystem.B.dylib' (no such file, no dyld cache)
```

### Key Error Messages

1. **`dyld cache '(null)' not loaded`** - The dynamic linker shared cache is corrupted or missing
2. **`syscall to map cache into shared region failed`** - System call to load the cache is failing
3. **`Library not loaded: /usr/lib/libSystem.B.dylib`** - Cannot find the core system library

## Root Cause

### System Details
- **OS:** macOS Sequoia 15.6.1 (Build 24G90)
- **Architecture:** arm64 (Apple Silicon)
- **Compiler:** GNU Fortran (Homebrew GCC 15.2.0)

### The Problem

macOS 15 (Sequoia) moved system libraries from `/usr/lib/` to the SDK at `/Library/Developer/CommandLineTools/SDKs/MacOSX.sdk/usr/lib/`, and the dynamic linker cache that maps these libraries is **corrupted or missing** on this system.

This is NOT a problem with:
- ❌ The KRC Fortran code
- ❌ The Python wrapper (`pykrc`)
- ❌ The build process
- ❌ Library linking
- ❌ Input file formatting
- ❌ The davinci interface

This IS a problem with:
- ✅ The macOS system dynamic linker cache
- ✅ Outdated system components
- ✅ Outdated Command Line Tools

---

## Why This Causes SIGKILL

1. Process attempts to start (KRC binary or any dynamically linked program)
2. macOS `dyld` (dynamic linker) tries to load required system libraries
3. `dyld` fails to map its shared cache due to system error
4. `dyld` cannot find `/usr/lib/libSystem.B.dylib` (core system library)
5. Kernel terminates process with **SIGKILL (-9)** because it cannot initialize

This happens BEFORE the program even executes its first instruction, which is why:
- No error logs are generated by KRC
- No stack traces exist
- Both direct execution and subprocess calls fail
- Python wrappers see instant termination

## Code Changes Made (Future-Proofing)

Even though the root cause is a system issue, we updated the Makefile to be more robust for macOS 15+:

### [Makefile](Makefile) Changes

**Added SDK detection and library path configuration:**

```makefile
# macOS 15+ SDK path for system libraries
SDKROOT := $(shell xcrun --show-sdk-path)
ifneq ($(SDKROOT),)
    export SDKROOT
    LDFLAGS_SDK = -L$(SDKROOT)/usr/lib
else
    LDFLAGS_SDK =
endif
```

**Updated all link targets to use SDK path:**

```makefile
# Before:
krc: $(OBJ8) $(CLIB)
	$(FC) $(LIBDIRS_PROD) -o $@ $(OBJ8) $(CLIB) $(SYSLIBS)

# After:
krc: $(OBJ8) $(CLIB)
	$(FC) $(LDFLAGS_SDK) $(LIBDIRS_PROD) -o $@ $(OBJ8) $(CLIB) $(SYSLIBS)
```

This ensures the linker knows where to find system libraries on macOS 15+, even though it won't fix the broken dyld cache.

## Lessons Learned

### What SIGKILL -9 Can Mean

- **Not always a code bug** - System-level issues can cause SIGKILL
- **Before the program runs** - If killed during dynamic linking, no logs exist
- **Affects all execution methods** - Direct, subprocess, fork/exec all fail the same way
- **System diagnostic required** - `otool -L`, `dyld` errors are key indicators

### Debugging Strategy

1. ✅ **Test the simplest case first** - Run the binary directly
2. ✅ **Check library dependencies** - Use `otool -L` on macOS, `ldd` on Linux
3. ✅ **Look for dyld errors** - Run binary and check stderr for linker messages
4. ✅ **Compare with known-good interface** - We used davinci as reference
5. ✅ **Test at multiple levels** - Binary, official interface, custom wrapper

### Why This Was Hard to Diagnose

- SIGKILL is silent - no error messages from the killed process
- Python subprocess shows instant termination with -9 exit code
- The error happens in system space, not user code
- Similar symptoms to memory issues, permission problems, or code bugs

## UPDATE: October 7, 2025 - ACTUAL ROOT CAUSE FOUND ✅

After system updates to macOS 26.0.1, the issue persisted. Further investigation revealed the **true root cause**:

### The Real Problem: 4GB Static Array

KRC contains a 4.4 GB static `__DATA` segment due to this line in `src/krcc8m.f:23`:

```fortran
PARAMETER (KOMMON=512000000) ! 512 million 8-byte words = 4 GB
```

**Apple Silicon cannot map 4GB static arrays** - the dynamic linker fails because there's insufficient address space to load both the huge static data and the required shared libraries.

### The Fix (CONFIRMED WORKING)

**Temporary solution:**
```fortran
PARAMETER (KOMMON=10000000) ! Reduced to 80 MB (10M * 8 bytes)
```

**Result:**
- Binary `__DATA` size: 4.4 GB → 400 MB
- Binary total size: 8.7 GB → 4.7 GB
- KRC now **runs successfully** on Apple Silicon!
- No more dyld errors!
- No more SIGKILL!

**Test output:**
```
$ cd run && ../krc < ../krc_python/pykrc/data/krc_support/fake_krc344
 This is KRC:  KRCv3.6.5            365
 DAYTIM = 2025 Oct  7 14:43:10
 [KRC running normally...]
```
